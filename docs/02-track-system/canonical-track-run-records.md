# Canonical Track Run Records

## Purpose

A **Track Run Record** is a standardized JSON execution record produced by every Track run, regardless of executor type. It serves as the shared execution language between the Benchmark Lab and the LocAIly runtime, enabling the Track Learning Evidence Loop.

## Record vs. Qualification vs. Track Definition vs. Benchmark Evidence

| Concept | Description | When Created | Consumed By |
|---|---|---|---|
| **Track Definition** | Declares inputs, outputs, steps, model roles, and validation rules for a Track. | Authored by developers when creating a Track. | Track runner, orchestrator, plan builder. |
| **Qualification Record** | Evidence that a capability (model, tool, worker contract, hardware profile) is approved or suitable for a Track. | Generated by Benchmark Lab from promoted benchmark evidence. | Local Brain model router, routing decisions. |
| **Track Run Record** | Evidence of one actual execution: what ran, who handled it, what happened, how long it took, whether validation passed. | Produced by every Track run (success or failure). | Audit, evidence loop, learning, debugging. |
| **Benchmark Evidence** | Promoted run summary from a controlled benchmark evaluation. | Approved and promoted through Benchmark Lab review workflow. | Qualification generation, report generation. |

### Key Distinction

- A **qualification record** says: "Capability X is approved for Track Y under these conditions."
- A **Track Run Record** says: "On this date/time, Track Y ran using capability X, and here is the result."
- One qualification record may support many Track Run Records.
- Many Track Run Records may contribute to future qualification decisions.

## Schema

The canonical schema is defined at `companion/evidence/schemas/track-run-record.schema.json` (version `locaily.track_run_record.v1`).

### Record Areas

| Area | Description |
|---|---|
| **Identity** | recordId, schemaVersion, trackId, workflowId, parentRunId, correlationId |
| **Timestamps** | createdAt (required), startedAt, completedAt |
| **Request** | requester, requestedCapability, inputSummary (summary-safe), inputRef, constraints |
| **Routing** | executorType (required), capabilityId, provider, relayNodeId, qualificationRecordId, routingReason, fallbackCandidates, shadowRecommendation (optional), enforcementDecision (optional) |
| **Execution** | status (required), durationMs, retryCount, fallbackUsed, plus executor-specific metadata (modelInfo, toolInfo, transformInfo, ruleInfo, relayNodeInfo) |
| **Output** | outputFormat, outputSummary (summary-safe), outputRef, structuredOutputValid, artifacts, evidenceRefs |
| **Validation** | status, validatorIds, failedChecks, warnings, score |
| **Performance** | totalDurationMs, modelLoadTimeMs, inferenceTimeMs, toolExecutionTimeMs, tokenCounts, ramVramMetrics |
| **Errors** | type, source, recoverable, fallbackResult, diagnosticSummary |
| **Child Runs** | Array of child Track Run Records for hybrid/multi-step workflows |

### Supported Executor Types

```txt
model       - Local or remote model runtime (Ollama, etc.)
tool        - Tool pack or built-in tool handler
transform   - Deterministic transformation (transform registry)
rule        - Rule-based executor
relay-node  - Remote capability via Relay Node protocol
hybrid      - Multi-step workflow combining executor types
```

### Parent-Child Relationships

Hybrid workflows produce a parent record with `executorType: "hybrid"` and one or more child records linked via `parentRunId`. Each child record describes one execution stage (e.g., model intent step, tool call, deterministic transform).

### Privacy and Safety

Records are summary-safe by default. Large raw inputs, full model outputs, binary evidence, and sensitive data must not be embedded directly. Use references (`inputRef`, `outputRef`, `artifacts`), checksums, and summaries instead.

## Record Builder

The builder at `companion/evidence/track-run-record-builder.js` provides functions:

- `buildTrackRunRecord(options)` — general builder
- `buildModelRecord(options)` — convenience for model-type records
- `buildToolRecord(options)` — convenience for tool-type records
- `buildTransformRecord(options)` — convenience for transform-type records
- `buildRuleRecord(options)` — convenience for rule-type records
- `buildRelayNodeRecord(options)` — convenience for relay-node-type records
- `buildHybridRecord(options)` — convenience for hybrid records
- `createRecordId(prefix)` — generates unique record IDs

All builders accept `childRuns` as an array of option objects to build nested records.

## Current Integrations

### Benchmark Lab Suite Runner (`benchmark-lab/engine/runners/suite-runner.js`)

Emits a Track Run Record after every mock or Ollama suite execution. The record captures model metadata, pass rates, validation results, and timing.

### Benchmark Lab Hybrid Deterministic Runner (`benchmark-lab/engine/runners/hybrid-deterministic-runner.js`)

Emits a parent Track Run Record with `executorType: "hybrid"` plus child records for each scenario trial's model, tool, and transform stages. Parent-child links are set via `parentRunId`.

## Runtime Integration (The Crew)

Every supported execution through The Crew now produces a canonical Track Run Record:

### Direct Track Execution (`/tracks/run`)

The orchestrator (`companion/crew/orchestrator.js`) emits one parent record with per-step child records. Supported tracks:
- `website_audit.lighthouse_handoff` — 7 steps (tool + model executors)
- `marketplace.dealsniper` — 3 steps (tool executors)

### Workflow Execution (`/workflows/run`)

The plan executor (`companion/orchestration/run-plan-executor.js`) emits a parent workflow record with per-step child records for the executed track.

### Failure Records

Failed executions emit records via `recordFailedExecution()` with `status: "failure"`, error diagnostics, and `validation.status: "not_validated"` when execution fails before initialization.

### Endpoint Evidence References

Both `/tracks/run` and `/workflows/run` return additive evidence fields without changing existing response structure:

```json
{
  "ok": true,
  "tool": "track-orchestrator",
  "task": "website_audit.lighthouse_handoff",
  "result": { ... },
  "meta": { ... },
  "evidence": {
    "trackRunRecordId": "track-abc123-def456",
    "childRecordIds": ["step-tool-...", "step-model-...", ...],
    "trackRunRecordRef": "data/evidence/track-run-records/track-abc123-def456.json",
    "warning": null
  }
}
```

### Audit Linkage

Track Run Records use `correlationId` (set to the request's `run_id`) for audit linkage. Audit records share the same `run_id`. This allows resolution without modifying the audit schema or duplicating record content.

### Persistence

Records are stored at `data/evidence/track-run-records/<recordId>.json` using the append-only store (`companion/evidence/track-run-record-store.js`). The store:
- Validates against `locaily.track_run_record.v1` before writing
- Rejects overwrites of existing record IDs
- Supports lookup by `recordId`, `workflowId`, `correlationId`, and `parentRunId`
- Strips null optional fields (the schema does not accept `null` for string fields)

### Recorder Service

The shared recorder (`companion/crew/runtime-track-run-recorder.js`) provides:
- `recordDirectTrackRun()` — for direct track execution
- `recordWorkflowRun()` — for workflow execution with parent-child relationships
- `recordFailedExecution()` — for failed executions

Callers pass `recordOpts` with provider info; the recorder handles builder creation, schema validation, storage, and audit reference threading.

## Future Integration Patterns

When emitting a Track Run Record from a new executor (beyond the built-in runtime paths):

1. Call the appropriate recorder function from `companion/crew/runtime-track-run-recorder.js`.
2. Or use the builder directly from `companion/evidence/track-run-record-builder.js`.
3. For child steps in a multi-step workflow, ensure `parentRunId` points to the parent record.
4. Persist via `companion/evidence/track-run-record-store.js`.
5. Ensure the record validates against the canonical schema.

Example for a tool execution:

```js
const { buildToolRecord } = require("./companion/evidence/track-run-record-builder");
const record = buildToolRecord({
  trackId: "my.track",
  capabilityId: "my-tool",
  provider: "tool-registry",
  status: "success",
  durationMs: 45,
  toolInfo: { toolId: "my-tool", task: "process", version: "1.0.0" },
  output: { outputSummary: "processed 3 items" }
});
```

## Enforcement Decision

When guarded qualification-aware routing is evaluated, the `routing.enforcementDecision` field is populated in the Track Run Record. This is an optional, additive field — records without it remain valid.

### Structure

```json
{
  "state": "enforced",
  "eligible": true,
  "attempted": true,
  "applied": true,
  "reason": "All enforcement gates passed",
  "failedConditions": [],
  "overrideApplied": false,
  "fallbackCapabilityId": "original-model",
  "fallbackTriggered": false,
  "fallbackSucceeded": false,
  "originalError": null,
  "selectedCapabilityId": "original-model",
  "recommendedCapabilityId": "qualified-model",
  "executedCapabilityId": "qualified-model",
  "qualificationRecordId": "qual-rec-001"
}
```

### Fields

| Field | Type | Description |
|---|---|---|
| `state` | string | Enforcement state at time of decision (`disabled`, `shadow`, `eligible`, `enforced`, `suspended`) |
| `eligible` | boolean | Whether the recommendation was eligible for enforcement |
| `attempted` | boolean | Whether enforcement evaluation was attempted |
| `applied` | boolean | Whether enforcement replaced the current capability with the recommended one |
| `reason` | string | Human-readable reason for the final decision |
| `failedConditions` | array | Conditions that prevented enforcement (present when `applied` is `false`) |
| `overrideApplied` | boolean | Whether an active override blocked enforcement |
| `fallbackCapabilityId` | string | Capability used as fallback when enforced execution failed |
| `fallbackTriggered` | boolean | Whether fallback execution occurred |
| `fallbackSucceeded` | boolean | Whether fallback execution succeeded |
| `originalError` | object | Original error when enforced execution failed (message + code) |
| `selectedCapabilityId` | string | Original capability selected before enforcement |
| `recommendedCapabilityId` | string | Capability recommended by qualification |
| `executedCapabilityId` | string | Final capability that executed (may differ from recommended after fallback) |
| `qualificationRecordId` | string | Qualification record that supported the recommendation |

### When Enforcement Is Not Evaluated

The `enforcementDecision` field is omitted entirely when no enforcement policy is available or no recommendation exists. This includes:
- Tool-only track executions
- Model steps where the enforcement policy is not configured
- Runs before enforcement integration was added (no backward compatibility concerns)

### Fallback Behavior

When an enforced capability fails execution:

1. The original error is captured in `originalError`.
2. Execution is retried with the original selected model (`fallbackCapabilityId`).
3. `fallbackTriggered` is set to `true`.
4. `fallbackSucceeded` is set based on the fallback result.
5. `executedCapabilityId` is updated to reflect the actual executing capability.
6. No indefinite retry. No automatic qualification status modification.

### Routing Sequence

```txt
current selection
→ shadow recommendation
→ enforcement policy evaluation
→ final selection (enforced or original)
→ execution
→ fallback only if enforced capability fails
→ record with enforcement decision
```
